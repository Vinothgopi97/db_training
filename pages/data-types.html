MySQL Data Types — Detailed Guide

This page describes MySQL's common data types and practical guidance for choosing them. Examples use InnoDB (the default transactional engine in modern MySQL).

Numeric types

MySQL provides several integer and fixed/floating point numeric types. Choose integers for whole numbers and DECIMAL for exact monetary values.

  • TINYINT (1 byte) — range: -128 to 127 (signed) or 0 to 255 (UNSIGNED). Useful for small counters or boolean-like values. Example: TINYINT(1) BOOL is often used for boolean flags.
  • SMALLINT (2 bytes) — range: -32768 to 32767 (signed).
  • MEDIUMINT (3 bytes) — range: -8,388,608 to 8,388,607 (signed).
  • INT / INTEGER (4 bytes) — the common integer type: -2,147,483,648 to 2,147,483,647 (signed). Use INT UNSIGNED when you need only non-negative values and extra positive range.
  • BIGINT (8 bytes) — for very large integers (e.g., 64-bit IDs).
  • BIT(n) — bit-field of up to 64 bits; useful for compact flags.
  • DECIMAL(p,s) — exact fixed-point numeric. Use for money: DECIMAL(10,2) stores values up to 99999999.99 exactly without floating rounding errors.
  • FLOAT and DOUBLE — approximate floating-point types. Not suitable for monetary values when exactness is required.

Numeric examples

CREATE TABLE products ( id INT AUTO_INCREMENT PRIMARY KEY, price DECIMAL(10,2) NOT NULL, stock INT UNSIGNED DEFAULT 0 );

Character and string types

Choose between fixed-length and variable-length character types based on expected usage and storage patterns.

  • CHAR(n) — fixed-length; padded with spaces to length n. Good for small, predictable sizes (e.g., country codes).
  • VARCHAR(n) — variable-length string with maximum n. Widely used for names, emails, etc.
  • TEXT types — TINYTEXT, TEXT, MEDIUMTEXT, LONGTEXT — for long unstructured text. Storage and max length increase from TINY -> LONG.
  • BINARY / VARBINARY — fixed/variable-length binary strings for non-text bytes.
  • BLOB types — TINYBLOB, BLOB, MEDIUMBLOB, LONGBLOB — store binary large objects (files, images, etc.).
  • ENUM — enumeration of allowed string values. Efficient and enforces a limited set of choices. Example: ENUM('small','medium','large').
  • SET — bitset of predefined string values (multiple values can be stored).

String examples

CREATE TABLE users ( id INT AUTO_INCREMENT PRIMARY KEY, username VARCHAR(50) NOT NULL, bio TEXT, avatar VARBINARY(1024) ); -- enum example CREATE TABLE shirts (id INT AUTO_INCREMENT PRIMARY KEY, size ENUM('XS','S','M','L','XL'));

Date and time types

MySQL supports a range of date/time types. Use DATETIME when you want a timezone-independent timestamp, TIMESTAMP when you want automatic timezone/UTC handling and automatic initialization/updates (server timezone aware).

  • DATE — 'YYYY-MM-DD'
  • TIME — 'HH:MM:SS' (time of day)
  • DATETIME — 'YYYY-MM-DD HH:MM:SS' (stores absolute date+time, no timezone conversion)
  • TIMESTAMP — similar to DATETIME but stored as UTC and converted to the session time zone on retrieval; has automatic behaviors for DEFAULT/ON UPDATE in some MySQL versions
  • YEAR — year in 2- or 4-digit form

Date/time example

CREATE TABLE events ( id INT AUTO_INCREMENT PRIMARY KEY, start_at DATETIME NOT NULL, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );

JSON and semi-structured data

MySQL has a native JSON type that stores JSON documents efficiently and supports JSON functions and indexing via generated columns. Use JSON for flexible, schemaless pieces of data.

CREATE TABLE orders ( id INT AUTO_INCREMENT PRIMARY KEY, metadata JSON );

Spatial (GIS) types

MySQL supports spatial types (GEOMETRY, POINT, LINESTRING, POLYGON, etc.) and spatial indexes. Use these for mapping, location queries and GIS workloads.

Binary / UUID storage

MySQL doesn't have a dedicated UUID type; common patterns are storing UUIDs as CHAR(36) (text) or as BINARY(16) (compact). Use BINARY(16) for storage efficiency and index size benefits (store raw bytes from UUID).

Choosing types & best practices

  • Use the smallest integer type that fits your range. Prefer UNSIGNED when values are non-negative.
  • Use DECIMAL for monetary values to avoid floating-point rounding.
  • Prefer VARCHAR over TEXT when you need to index or constrain length; TEXT types have limitations on indexing unless using full-text indexes.
  • Use JSON when you need flexible nested structures, but prefer normalized columns for frequently queried attributes (or extract them into generated columns for indexing).
  • Avoid storing arrays as a single delimited string — prefer junction tables or JSON depending on use-case.
  • For high-write workloads, be cautious with very wide rows (many large TEXT/BLOB columns) as they affect IO and page splits.

Storage considerations:

  • Character set (utf8mb4) and collation affect storage size for string types; utf8mb4 uses up to 4 bytes per character.
  • Indexes on long VARCHAR/TEXT columns may need length prefixes or generated columns.
  • Use appropriate index types (B-tree for most columns; FULLTEXT for text search; SPATIAL for geometry with MyISAM or InnoDB spatial indexes on newer MySQL).
